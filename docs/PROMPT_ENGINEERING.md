# Prompt Engineering: Hype Detector Case Study

## The Problem

The Hype Detector tab renders a custom HTML dashboard generated by Claude — ticker cards with sentiment scores, hype indicators, and visual heat signals. The dashboard is rendered directly in the browser. Claude's output must be valid HTML every time.

It was not.

Early builds produced inconsistent output across runs:

**Run A (correct):**
```
<div class="hype-dashboard">
  <div class="ticker-card" data-ticker="NVDA">
    ...
  </div>
</div>
```

**Run B (prose):**
```
Here's an analysis of current market hype levels for your watchlist...
NVDA is showing elevated sentiment with strong retail interest...
```

**Run C (fenced markdown):**
```html
<div class="hype-dashboard">
  ...
</div>
```

**Run D (mixed):**
```
Based on current market data, here is the hype dashboard:
<div class="hype-dashboard">
  <div class="ticker-card"...
```

Runs B, C, and D all caused silent rendering failures. The tab displayed blank or broken content with no error and no indication to the user that anything went wrong.

---

## Why This Happens

Claude is trained to be helpful and conversational. Without explicit constraints, it defaults to the output format that feels most natural for a given phrasing — and "natural" shifts based on subtle differences in wording, context, and session state. This is not a bug. It is the default behavior of a conversational model operating without a strict output contract.

---

## The Fix

The system prompt was rewritten as an explicit output contract with four components:

### 1. Format Specification
The opening instruction states the output requirement in absolute terms:

> "Your entire response must be valid HTML. It must start with `<div` and end with `</div>`. Do not include markdown. Do not include prose. Do not include code fences. Do not explain what you are doing. Return only the HTML."

### 2. Structure Specification
The exact HTML structure is specified — wrapper element, card structure, required CSS classes, required data attributes. Claude is given the schema, not just the concept.

### 3. Negative Examples
Each disallowed output format is shown explicitly with a label:

> "Do not do this: `Here is the hype dashboard for your watchlist...`
> Do not do this: ` ```html <div... ``` `
> Do not do this: `Based on the data, here is the dashboard: <div...`"

Negative examples anchor the model away from the formats it naturally defaults to.

### 4. Output Anchor
The prompt ends at the opening of the expected output:

> "Return only: `<div class=\"hype-dashboard\">`"

Starting the expected output in the prompt primes Claude to continue in that format. This is the single most effective technique — the model completes the pattern rather than choosing a format.

---

## Result

Output is now deterministic across runs. The frontend renders the Hype Detector tab reliably. No retry logic. No format validation. No fallback handling.

---

## Generalizable Principles

| Technique | When to Use |
|---|---|
| Absolute format instruction | Any time output format must be exact |
| Negative examples | When the model defaults to formats you do not want |
| Output anchor | When the output must start with a specific string |
| Structure specification | When the output has a required schema |
| Prompt versioning in DB | When prompts need to evolve without redeployment |

Non-determinism in LLM output is almost always a prompt engineering problem. Retry logic and post-processing are the wrong fix — they treat the symptom. A well-designed prompt contract treats the cause.
